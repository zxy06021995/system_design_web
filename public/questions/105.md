# 三、开场第一分钟怎么说

先不要急着画 Kafka。

先说目标。

你可以这样开场：

> 我先定义一下系统目标。
> 这个系统要处理订单主链路，比如锁库存、扣钱、确认订单。
> 它还要处理辅助任务，比如发邮件、加积分、埋点。
> 核心要求是：高并发下系统要稳，worker crash 后任务不能丢，消息重复时不能重复扣钱，最终订单状态必须正确。

然后补一句：

> 我会先讲整体架构，再讲消息队列和 pipeline 怎么推进，最后讲失败恢复、重复消费、顺序和扩展性。

这段非常像面试回答。

# 四、第一张图怎么画

先画最基础的图：

```text
Client
  ↓
API Server
  ↓
Database
  ↓
Message Queue
  ↓
Workers
```

然后说：

> API 层负责快速接请求和落库，真正慢的动作放到 queue 后面异步做。
> 这样可以削峰、解耦、提高稳定性。

这一步很重要。
因为很多候选人一上来就讲 partition，反而没讲清为什么要有 queue。

# 五、先定义系统成功标准

这是面试里特别加分的一步。

你可以说：

> 订单系统最终只允许两种正确结果：

```text
钱扣了 + 库存扣了 + 订单成功
```

或者：

```text
钱没扣 + 库存没扣 + 订单取消
```

然后补一句：

> 允许中间短暂不一致，但最终必须一致。

这就是：

```text
最终一致性
```

这句话很重要。

---

# 六、订单主链路怎么设计

你可以先讲最简单的流程：

```text
create_order(PENDING)
→ reserve_stock
→ capture_payment
→ confirm_order
```

然后说：

> 我不会在创建订单时把所有任务一次性都发出去。
> 我会采用 pipeline 的方式，前一步成功后，再推进下一步。

这是关键点。

---

# 七、为什么不能一开始生成所有任务

你可以主动补一句：

> 如果一开始就把 reserve_stock、capture_payment、send_email 全都丢进 queue，
> 那么一旦库存失败，后面的扣钱任务可能还是会执行，系统就会出现错误状态。

所以正确做法是：

```text
reserve_stock success
→ emit capture_payment

capture_payment success
→ update order CONFIRMED
→ emit send_email
```

这就是 pipeline。

---

# 八、主链路和辅助任务怎么拆

这一步面试官很喜欢追问。

你可以这样答：

> 我会把主链路和辅助任务分开。
> 主链路包括锁库存、扣款、确认订单。
> 辅助任务包括邮件、短信、积分、埋点。
> 主链路必须成功，辅助任务失败可以补偿。

然后画：

```text
order_topic:
  reserve_stock
  capture_payment
  confirm_order

notification_topic:
  send_email
  send_sms

analytics_topic:
  track_event
```

然后说：

> 这样邮件系统慢，不会拖住扣钱。

这句很好用。

---

# 九、Message Queue 为什么存在

这里用最简单的逻辑讲：

> Queue 的作用有三个：
> 第一，削峰，把瞬时洪水变成平滑处理。
> 第二，解耦，订单系统不用同步等待邮件系统。
> 第三，异步，API 可以快速返回。

如果面试官问高并发，你就说：

> 比如突然来 10 万订单，API 先接住，worker 后台慢慢消费，而不是让数据库和支付系统同时被冲爆。

# 十、Topic 和 Partition 怎么讲

这部分不要一开始太深。

先说：

> Topic 是消息分类，比如订单、通知、支付。
> Partition 是 topic 里的多个并行队列，用来提高吞吐量。

然后画：

```text
order_topic
  ├─ partition 0
  ├─ partition 1
  ├─ partition 2
  └─ partition 3
```

然后补一句：

> 多个 partition 让多个 worker 可以并行处理不同消息。

---

# 十一、Partition Key 怎么选

这是很容易被问到的。

你可以这样答：

> Partition key 要按系统真正关心的顺序单位来选。
> 订单系统通常按 order_id。
> 用户消息系统可以按 user_id。
> 库存系统往往按 product_shard，而不是直接按 product_id。

如果面试官问为什么订单系统按 order_id：

> 因为我要保证同一个订单的 reserve_stock、capture_payment、confirm_order 进入同一个 partition，保持局部顺序。

这句很重要。

---

# 十二、顺序怎么保证

标准答法：

> MQ 通常只保证 partition 内顺序，不保证全局顺序。
> 真实系统也不需要全局顺序，只需要同一个订单内部顺序。

然后你可以补一句：

> 全局顺序代价太高，会严重影响吞吐量和扩展性，所以通常选择局部顺序。

这是面试很爱听的话。

---

# 十三、什么是 Hot Partition，怎么答

如果面试官问：

```text
如果某个 key 特别热怎么办？
```

你可以说：

> 如果 key 选得不好，比如库存系统直接用 product_id，热门商品会把大量消息打到同一个 partition，形成 hot partition。
> 这样一个 worker 很忙，其他 worker 很闲。

解决办法讲两个就够了：

```text
1. 换更均匀的 key
2. 做 shard，比如 product_shard
```

例如：

```text
iphone_stock_1
iphone_stock_2
iphone_stock_3
```

---

# 十四、消息重复怎么办

这是必问题。

你一定要说：

> MQ 系统一般默认消息可能重复，所以消费侧必须幂等。

然后讲人话版本：

> 同一条消息执行 1 次和执行 10 次，结果要一样。

实现方式你讲两个：

```text
1. request_id / idempotency key
2. 数据库唯一约束 / action 表
```

例如：

```text
order123_capture_payment
```

支付系统收到相同 request_id，只扣一次。

---

# 十五、order_id 和 request_id 的区别怎么讲

这也是高频点。

你可以直接说：

> order_id 表示是哪一个订单。
> request_id 表示这个订单的某一个具体外部动作。
> 所以 request_id 通常会包含 order_id + action_type。

比如：

```text
order_123_reserve_stock
order_123_capture_payment
order_123_refund_payment
```

然后补一句：

> 同一个动作的重试必须复用同一个 request_id，不能每次重试都生成新的。

这句很关键。

---

# 十六、Worker crash 怎么办

这是核心中的核心。

你可以这样答：

> Worker crash 是常态，不是例外。
> 所以系统必须把每个动作做成可恢复的。

常见机制：

```text
heartbeat
timeout
retry
status machine
```

然后讲最简单流程：

```text
PENDING
RUNNING
SUCCESS
FAILED
UNKNOWN
```

解释时这样说：

- `PENDING`：还没做
- `RUNNING`：正在做
- `SUCCESS`：成功
- `FAILED`：明确失败
- `UNKNOWN`：不确定外部有没有成功

---

# 十七、为什么需要 UNKNOWN

这句特别容易加分。

你可以说：

> 如果 worker 调支付时超时了，我不能立刻判定失败。
> 因为钱可能已经扣了，只是结果没返回。
> 这种情况我会把动作标成 UNKNOWN，然后通过 request_id 去外部系统查询真实状态。

这就是成熟系统设计。

---

# 十八、ACK 什么时候发

如果问消费确认机制，你这样答：

> ACK 不能太早，也不能太晚。
> 太早会导致任务丢失。
> 太晚会导致消息重复投递。
> 所以系统通常接受“可能重复但不丢”，再用幂等保证安全。

这是很标准的答案。

---

# 十九、At least once / Exactly once 怎么答

最稳的答法是：

> 真实系统里最常见的是 at-least-once + idempotency。
> 因为 exactly-once 在跨服务、跨数据库、跨外部系统时非常难，也非常贵。
> 所以工程上更常见的做法是允许重试和重复，再通过 request_id 和状态机把结果收敛正确。

这段很适合面试。

---

# 二十、Retry 怎么设计

你可以说：

> Retry 不能无脑重试。
> 我会用 exponential backoff 加 jitter。

如果面试官不喜欢术语，你就说：

```text
1秒
2秒
4秒
8秒
```

再加一点随机：

```text
1.2秒
2.3秒
4.1秒
```

为什么？

> 防止所有 worker 同时重试，把下游彻底打爆。

---

# 二十一、Rate limit 怎么讲

这一步可选，但很加分。

> 对支付、库存这类关键下游，我会做 rate limit。
> 比如每秒最多只允许 100 个扣款请求出去。
> 否则下游一慢，所有 worker 同时冲上去，系统会雪崩。

---

# 二十二、数据库表怎么讲

面试不用讲太细，但要有两张表的概念。

### 1. orders 表

```text
order_id
status
created_at
updated_at
```

状态：

```text
PENDING
CONFIRMED
FAILED
CANCELLED
```

### 2. actions 表

```text
order_id
action_type
status
request_id
owner
heartbeat
```

主键：

```text
(order_id, action_type)
```

作用：

> 记录每个动作做到哪一步，并保证同一个动作最多成功一次。

# 二十三、模拟完整回答框架

如果面试官让你从头讲，你可以按这个顺序说：

## 第一步：定义目标

> 这个系统要处理订单主链路和异步辅助任务。
> 要求高并发下稳定，消息不丢，重复消费安全，最终状态一致。

## 第二步：整体架构

> API 接请求后先创建订单壳子，写数据库，再发消息到 queue。
> worker 后台异步执行 reserve_stock、capture_payment 等动作。

## 第三步：pipeline

> 不会一开始创建所有任务。
> 而是前一步成功后，才推进下一步。
> reserve_stock 成功后才发 capture_payment，扣款成功后再确认订单和发通知。

## 第四步：topic 和 key

> 主链路和辅助任务分 topic。
> 订单主链路按 order_id 分区，保证同一订单局部顺序。
> 通知系统单独 topic，避免拖慢主链路。

## 第五步：幂等和重复消息

> MQ 允许重复投递，所以消费侧要幂等。
> 对外部副作用使用 request_id，例如 order123_capture_payment，确保支付最多执行一次。

## 第六步：失败恢复

> worker crash 后通过 heartbeat 和 timeout 让其他 worker 接管。
> 如果外部动作结果不确定，标 UNKNOWN，再用 request_id 去查询外部状态。

## 第七步：扩展性和稳定性

> 通过 partition 提升吞吐量，通过 backoff 和 rate limit 保护下游。
> 对热点 key 做 shard，避免 hot partition。

## 第八步：总结

> 整体设计思路是：
> 用 queue 解耦和削峰，用 pipeline 控制顺序推进，用 idempotency 解决重复，用状态机和恢复机制保证最终一致。

这就是一套完整面试答案。

# 二十四、面试官可能追问什么

下面这些几乎是必追问。

## 追问1：为什么不用数据库事务把所有步骤包起来？

答：

> 因为库存、支付、通知往往是不同服务，跨服务长事务代价太高，耦合重、性能差、可用性差。
> 所以更常用本地事务 + 消息 + 补偿，走最终一致性。

## 追问2：为什么不保证全局顺序？

答：

> 没必要，而且代价太大。
> 系统通常只需要同一个订单内部顺序。
> 不同订单之间完全可以并行。

## 追问3：如果 send_email 消息发出后重复了怎么办？

答：

> 邮件消费方要做幂等，比如基于 request_id 去重。
> 或者即使允许偶发重复，也要有上限和保护，不能无限重复。

## 追问4：如果 capture_payment 成功了，但更新订单状态失败了怎么办？

答：

> 动作会进入 UNKNOWN 或部分完成状态。
> 恢复流程会根据 request_id 去支付系统确认，再把订单状态补写正确。

## 追问5：如果某个 partition 很慢怎么办？

答：

> 这是 hot partition 或慢消息阻塞问题。
> 可以通过更均匀的 key、shard、拆分主链路和辅助任务、避免单条消息过重来优化。

# 二十五、你可以直接背的总结版

这是最适合面试最后 30 秒总结的一段：

> 我会把系统设计成一个基于 message queue 的异步 pipeline。
> API 层先落订单壳子，后台 worker 按步骤执行 reserve_stock、capture_payment、confirm_order。
> 主链路和通知等辅助任务分开，主链路按 order_id 做 partition key，保证同一订单局部顺序。
> 因为 MQ 可能重复投递，所以所有副作用都通过 request_id 做幂等。
> worker crash 时通过 heartbeat、timeout 和状态机恢复；如果外部结果不确定，就进入 UNKNOWN，再向外部系统查询。
> 整体目标是实现高并发下的稳定性、可恢复性和最终一致性。

这段已经很像成熟候选人的回答。
