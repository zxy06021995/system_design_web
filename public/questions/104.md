# Q104：Design Enterprise Knowledge GPT（企业知识文档 GPT，面试复述版）

## 1. 三句话题目本质
### 白话版
1. 这题像给公司做一个“只会回答公司资料的聪明图书管理员”。  
2. 员工问问题时，它不能乱编，只能从公司文档里找证据，再把答案和出处一起说出来。  
3. 最难的地方不是“让 AI 说话”，而是“让 AI 只看它该看的资料，而且文档一更新就马上用新资料回答”。  

### 升级版
1. 企业知识 GPT 的核心是 `RAG = 检索 + 权限过滤 + LLM 总结 + 引用输出`。  
2. 关键难点是双重权限校验、文档更新后的索引刷新、缓存失效、防幻觉和高峰成本控制。  
3. 面试高分要点：把查询链路、上传链路、权限一致性、降级路径、缓存策略和代码机制讲成闭环。  

## 2. 真实场景故事
### 白话版
公司里有很多资料：员工手册、报销制度、销售话术、合同模板。  
如果新同学问“差旅报销上限是多少”，他不想翻 200 页 PDF，他想直接问一句话。  
所以我们做一个“公司资料问答助手”，像图书馆管理员一样，先去找资料，再把答案和页码告诉他。  

### 升级版
一家 3 万人的公司把制度文档、HR FAQ、销售 SOP、产品说明书都放进知识平台。高峰期每天有 80 万次提问，文档每天更新几千次，且不同部门权限不同。  
如果系统只做向量检索，不做权限控制，可能把法务合同内容答给不该看的人；如果只做关键词检索，用户又搜不到语义相近的答案。  
最终方案是：`Hybrid Search + 权限白名单 + LLM 受限总结 + 引用回查 + MQ 异步更新索引 + 按版本失效缓存`。  

## 3. 术语白话表（>=10）
### 白话版
把下面这些词先当成“工具箱”，面试时你会一直用到。  

### 升级版
| 术语 | 白话解释 | 面试可复述 |
|---|---|---|
| RAG | 先找资料，再让模型回答 | 不是让模型凭空想，而是先喂证据 |
| Chunk | 文档小片段 | 一大页资料要切成很多小块才好检索 |
| Embedding | 向量表示 | 把一句话变成机器能比较相似度的数字 |
| Hybrid Search | 混合检索 | 关键词检索 + 向量检索一起用 |
| Citation | 引用来源 | 答案后面必须带出处 |
| ACL | 权限控制列表 | 谁能看哪份文档 |
| OCR | 文字识别 | 图片/PDF 扫描件先识别成文字 |
| Index Version | 索引版本号 | 文档更新后版本变化，旧缓存失效 |
| Re-rank | 重排序 | 先找候选，再按质量重排 |
| Cache | 缓存 | 热门问题直接复用上次结果 |
| MQ | 消息队列 | 上传后慢处理任务异步做 |
| Answer Validation | 答案校验 | 检查回答有没有脱离证据胡说 |
| DLQ | 死信队列 | 多次处理失败的文档单独隔离 |
| Hot/Cold Storage | 冷热分层 | 新文档和热文档快存，老文档冷存 |

## 4. 需求澄清（功能/非功能/不做范围/SLO）
### 白话版
先别急着画大图，先回答四个小问题：  
1. 用户要做什么？  
2. 系统要多快？  
3. 哪些事一定不能错？  
4. 第一版不做什么？  

### 升级版
#### 4.1 功能需求
- 用户自然语言提问，系统返回答案。  
- 返回答案必须带引用来源，至少包含 `docId/chunkId/标题`。  
- 支持上传 `PDF / 图片 / Excel`。  
- 支持文档内容更新和权限更新。  
- 支持查询文档索引状态、失败重试、重新建索引。  

#### 4.2 非功能需求
- 高可用：Query Service、Index Worker、Search Cluster 多实例部署。  
- 低延迟：问答接口 `P95 < 5s`。  
- 权限安全：权限修改后的授权判断必须强一致。  
- 可扩展：支持文档量和用户量增长。  
- 可控成本：减少不必要的 LLM 调用和长上下文。  

#### 4.3 不做范围
- 不做通用互联网搜索。  
- 不做多轮 Agent 工作流编排。  
- 不做训练企业私有大模型，优先用现成 LLM + RAG。  

#### 4.4 SLO
- `ask_latency_p95 < 5s`  
- `permission_update_visibility < 1s`（授权判断）  
- `index_update_p95 < 3min`（文档更新到可检索）  
- `citation_missing_rate < 0.1%`  

## 5. 容量估算（数字推导）
- 公司文档数：500 万份。  
- 平均每份文档切成 40 个 chunk，总 chunk 约 2 亿。  
- 每个 chunk 平均 600B 原文 + 1536 维向量。  
- 日新增或更新文档：5 万份，约 200 万个 chunk 需要重建。  
- 平峰提问 QPS 120，峰值 QPS 800。  
- 热门问题缓存命中目标：`35%~50%`。  
- 向量检索每次先取 top 50，再权限过滤和重排后给 LLM top 8~12。  
- 热数据：近 30 天更新文档放热索引；历史低频文档走冷存和低成本分片。

- 向量索引：按 `doc_id hash` 分片，支持 ANN 检索。  
- 关键词索引：`term -> posting list`。

- 权限源表索引：`(doc_id, group_id)`、`(group_id, doc_id)`。  

## 9. 核心流程（正常/高峰/故障恢复）
### 白话版
正常时像这样：  
1. 先看有没有现成答案。  
2. 没有就去找资料。  
3. 找到资料再让 AI 组织成一句通顺的话。  
4. 最后再检查一次“这些资料你能不能看”。  

### 升级版
#### 9.1 正常问答流程
1. Query Service 计算缓存 key：`hash(question + user_group + index_version)`。  
2. 命中缓存时，先检查缓存答案引用的 `docId` 是否仍可访问。  
3. 缓存未命中时，Hybrid Search 执行关键词检索 + 向量检索。  
4. 候选 chunk 先按权限源表白名单过滤，再重排序。  
5. 把 topK chunk 和 prompt 发送给 LLM。  
6. LLM 输出答案和引用。  
7. Citation Validator 再校验一次引用权限和引用真实性。  
8. 通过后返回，并写入缓存。  

#### 9.2 文档上传与更新流程
1. 文档上传到对象存储和元数据 DB。  
2. 事务内写 `document`，再发 `document.changed` 到 MQ。  
3. Worker 执行：解析 -> OCR -> chunk -> embedding -> 更新索引。  
4. 成功后提升 `index_version`。  
5. Cache Evictor 删除相关缓存。  

#### 9.3 高峰流程
1. 热门问题优先走缓存。  
2. Search Service 高峰时先缩小 topK 和上下文长度。  
3. LLM 资源紧张时切小模型 fallback。  
4. 极端高峰时只返回检索片段摘要，不走长回答。  

#### 9.4 故障恢复流程
1. OCR/embedding 失败进入重试队列。  
2. 多次失败进入 DLQ，人工重放。  
3. Search 服务不可用时降级到缓存 + 关键词简单查找。  
4. LLM 超时时返回“暂时无法回答”，不返回无证据内容。  

## 10. 一致性与事务边界
### 白话版
这题最容易错的地方是：  
“系统刚把权限收走，但 AI 还用旧资料回答了。”  
所以权限这件事要比搜索和向量更新更严格。  

### 升级版
- 事务边界 1：`document` 元数据写入和 MQ 事件写入必须同事务，保证不丢更新任务。  
- 事务边界 2：`permission_binding` 更新必须强一致提交，读请求立刻使用最新权限源表。  
- 事务边界 3：向量索引和关键词索引允许异步更新，但不能成为最终授权依据。  
- 双重权限校验：
  - 搜索阶段：候选 chunk 经过权限源表过滤。  
  - 返回阶段：对 LLM 输出的所有 `docId/chunkId` 再次校验。  
- 结论：检索系统可以最终一致，权限判断必须以强一致源表为准。  

## 11. 可用性与容错（含 RTO/RPO）
- Query Service、Search API、LLM Gateway 多副本部署。  
- MQ 多副本，文档索引任务异步重试。  
- 文档原始文件放对象存储，多 AZ 冗余。  
- `RTO = 15 分钟`：检索集群单区故障后恢复。  
- `RPO = 0`：文档元数据与权限更新不能丢。  
- OCR、Embedding、Index Writer 单组件失败不会阻塞整个查询链路，只影响新文档可检索时间。  

## 12. 可观测性（指标+阈值+处置动作）
### 白话版
可观测性就像给系统装体温计和报警器。  
如果它“发烧”了，你要知道哪里热、为什么热、怎么退烧。  

### 升级版
- `ask_latency_p95 > 5000ms` 持续 5 分钟：扩 Query Service，缩小 topK，启用短上下文。  
- `search_error_rate > 1%`：切换到降级检索路径。  
- `llm_timeout_rate > 3%`：启用小模型 fallback，限制长答案。  
- `citation_missing_rate > 0.1%`：立即阻断无引用答案下发。  
- `index_lag_minutes > 10`：扩容 Worker，优先处理热文档。  
- `permission_check_mismatch > 0`：P1 告警，暂停返回可疑答案。  

## 13. 安全与合规
- 认证：企业 SSO / JWT。  
- 授权：RBAC + `permission_binding` 强一致校验。  
- 存储：文件加密、向量索引和缓存不落明文敏感字段。  
- 审计：记录谁上传了文档、谁改了权限、谁问了高敏问题。  
- 合规：敏感文档支持“不进入 LLM，只返回检索片段”的策略。  

## 14. 成本与取舍
### 白话版
最贵的不是存文件，而是每次都去叫大模型认真读很多资料。  
所以系统要学会“能不叫大模型就不叫，能少给资料就少给”。  

### 升级版
- 取舍 1：回答更长更全，成本更高；第一版优先短答案 + 强引用。  
- 取舍 2：topK 越大召回越全，但 LLM token 成本和延迟都会上升。  
- 取舍 3：全量重建索引最稳，但更新成本高；优先增量更新 + 定期全量校验。  
- 降本策略：
  - 相同问题缓存  
  - 相似问题聚类预热  
  - 热门问题预生成答案  
  - 上下文压缩和 chunk 去重  
  - 小模型 answer validation，避免大模型二次调用

```java
        List<String> sortedGroups = new ArrayList<>(groups);
```

### 15.2 检索阶段权限过滤
```java
public class PermissionAwareRetriever {
    public List<ChunkCandidate> retrieve(String question, UserContext user, int topK) {
        List<ChunkCandidate> lexical = bm25Search.search(question, 50);
        List<ChunkCandidate> vector = vectorSearch.search(question, 50);
        Map<String, ChunkCandidate> merged = new LinkedHashMap<>();
        for (ChunkCandidate c : lexical) merged.put(c.getChunkId(), c);
        for (ChunkCandidate c : vector) merged.putIfAbsent(c.getChunkId(), c);

        Set<String> allowedDocIds = permissionRepo.findAllowedDocIds(user.getGroupIds());
        return merged.values().stream()
            .filter(c -> allowedDocIds.contains(c.getDocId()))
            .sorted(Comparator.comparingDouble(ChunkCandidate::getRankScore).reversed())
            .limit(topK)
            .toList();
    }
}
```

### 15.3 LLM 输出引用二次校验
```java
public class CitationDoubleChecker {
    public AnswerResult validate(UserContext user, AnswerDraft draft) {
        if (draft.getCitations().isEmpty()) {
            throw new BizException("RAG_422_NO_EVIDENCE", "answer must contain citations");
        }
        for (CitationRef ref : draft.getCitations()) {
            boolean allowed = permissionRepo.hasAccess(ref.getDocId(), user.getGroupIds());
            if (!allowed) {
                throw new BizException("RAG_403_DOC_FORBIDDEN", "citation not accessible");
            }
            boolean chunkExists = chunkRepo.exists(ref.getChunkId(), ref.getDocId());
            if (!chunkExists) {
                throw new BizException("RAG_422_NO_EVIDENCE", "citation chunk not found");
            }
        }
        return new AnswerResult(draft.getAnswer(), draft.getCitations());
    }
}
```

### 15.5 MQ Worker：解析 / OCR / chunk / embedding / 写索引
```java
public class DocumentIndexWorker {
    public void handle(DocumentChangedEvent event) {
        DocumentMeta meta = documentRepo.get(event.docId());
        ParsedDocument parsed = parserFactory.forType(meta.getParserType()).parse(objectStore.read(meta.getDocId()));
        String plainText = parsed.requiresOcr() ? ocrClient.extractText(parsed.getBinaryPages()) : parsed.getText();

        List<Chunk> chunks = chunker.split(plainText, 500, 80);
        List<EmbeddingRow> rows = new ArrayList<>();
        for (Chunk chunk : chunks) {
            float[] vector = embeddingClient.embed(chunk.content());
            rows.add(new EmbeddingRow(chunk.chunkId(), meta.getDocId(), vector, chunk.content()));
        }

        indexWriter.replaceDocument(meta.getDocId(), rows, meta.getLatestPermissionVersion());
        documentRepo.markIndexed(meta.getDocId(), chunks.size());
    }
}
```

### 15.6 答案校验与降级
```java
public class AnswerGuardService {
    public AskResponse guardOrFallback(AnswerDraft draft, List<ChunkCandidate> evidence, long elapsedMs) {
        boolean hasCitation = !draft.getCitations().isEmpty();
        boolean enoughEvidence = evidence.size() >= 2;
        if (hasCitation && enoughEvidence) {
            return AskResponse.success(draft.getAnswer(), draft.getCitations(), false, elapsedMs);
        }
        return AskResponse.degraded(
            "我现在找到了相关资料，但证据还不够稳妥。请查看下面引用片段。",
            evidence.stream().limit(3).map(ChunkCandidate::toCitation).toList(),
            elapsedMs
        );
    }
}
```

```javascript
      body: JSON.stringify({ question, userId, userGroup, conversationId: "conv-demo" })
    });

    if (!resp.ok) {
      setState("error");
      setMessage(`问答失败: ${resp.status}`);
      return;
    }

    const body = await resp.json();
    setAnswer(body.answer || "");
    setCitations(body.citations || []);
    setState("done");
    if (body.degraded) {
      setMessage("系统已降级，当前返回的是保守答案");
    }
  }

  return null;
}
```

### 16.2 文档上传 + 索引状态轮询
```javascript
import { useState } from "react";

export function DocumentUploadFlow() {
  const [state, setState] = useState("idle");
  const [docId, setDocId] = useState("");
  const [message, setMessage] = useState("");

  async function upload(payload) {
    setState("loading");
    const resp = await fetch("/api/v1/documents", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify(payload)
    });
    if (!resp.ok) {
      setState("error");
      setMessage("上传失败，请检查文件格式或权限");
      return;
    }

    const body = await resp.json();
    setDocId(body.docId);
    setState("polling");

    let tries = 0;
    while (tries < 20) {
      const statusResp = await fetch(`/api/v1/documents/${body.docId}/index-status`);
      if (statusResp.ok) {
        const statusBody = await statusResp.json();
        if (statusBody.status === "INDEXED") {
          setState("done");
          setMessage("文档已完成索引，可以开始问答");
          return;
        }
      }
      tries += 1;
      await new Promise((resolve) => setTimeout(resolve, 3000));
    }

    setState("error");
    setMessage("索引时间过长，系统可能正在高峰处理");
  }

  return null;
}
```

## 19. 面试追问 + 可复述回答
### 白话版
如果面试官问得很快，你可以先用最简单的话回答：  
1. 它先找资料，再让 AI 说。  
2. AI 不能自己编，必须带出处。  
3. 找资料前和返回前都要看权限。  

### 升级版
1. 为什么不能只做向量检索，不做关键词检索？  
回答：企业问答里很多词是制度名、产品名、缩写词，只做向量检索容易漏召回，Hybrid Search 更稳。  

2. 为什么需要双重权限校验？  
回答：检索阶段过滤是第一层；但 LLM 可能在拼接答案时引用了越权 chunk，所以返回前必须对引用再做一次强一致权限校验。  

3. 权限修改为什么要求强一致，而索引更新可以异步？  
回答：权限错误是安全事故；索引延迟只是新文档暂时搜不到。授权判断必须读强一致源表。  

4. 如何防止模型胡编？  
回答：只给白名单证据、强制引用、证据不足拒答、用 answer validator 检查答案是否脱离上下文。  

5. 这题最大的成本瓶颈在哪里？  
回答：LLM token 成本和长上下文延迟，所以要靠缓存、压缩上下文、热门问题预热和多模型 fallback 降本。  

## 20. 新手学习路线
### 白话版
你可以按“搭积木”的顺序学，不要一上来就学最难的：  
1. 先懂文档怎么切小块。  
2. 再懂怎么找最像的小块。  
3. 再懂怎么加权限。  
4. 再懂怎么让 AI 用这些小块回答。  
5. 最后再学缓存、更新和降级。  

### 升级版
#### 第 1 层：只学 chunk
- 目标：知道为什么长文档不能整篇塞给模型。  
- 练习：把一页文档按段落或 token 数切成多个 chunk。  

#### 第 2 层：只学 embedding
- 目标：知道“相似问题能找到相似片段”。  
- 练习：给几个 chunk 算向量，比较相似度。  

#### 第 3 层：学检索
- 目标：理解关键词检索和向量检索的区别。  
- 练习：同一个问题分别用 BM25 和向量检索找答案。  

#### 第 4 层：学权限
- 目标：明白为什么权限不能只靠索引。  
- 练习：模拟权限撤销后缓存命中但被拦截的场景。  

#### 第 5 层：学缓存和更新
- 目标：理解 `index_version` 为什么要进缓存 key。  
- 练习：更新一篇文档，验证旧答案是否失效。  

#### 第 6 层：学工程化
- 目标：掌握 MQ、OCR、重试、DLQ、降级、监控。  
- 练习：画出完整的上传链路和查询链路。
